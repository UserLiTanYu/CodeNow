# 码上记（CodeNow）项目代码质量与工程化优化方案

> 文档版本：v1.0 | 编写日期：2026-07-14 | 基于全面代码审查

---

## 一、引言

码上记（CodeNow）项目已完成全部基础功能和六个扩展功能的开发。为提升代码质量、消除安全隐患、改善工程化水平，对项目进行了全面的代码审查和架构评估。

本优化方案按 **P0（关键）→ P1（重要）→ P2（建议）** 三个优先级分类，覆盖目录结构、配置管理、代码规范、部署运维、可扩展性五个维度，共梳理出 **28 项** 待优化问题。

**核心目标：**
- 消除安全漏洞和运行时 Bug
- 统一代码规范，减少重复代码
- 提升部署可靠性和可维护性
- 为后续功能迭代和团队协作打下基础

## 三、详细分析与改进方案

### 3.1 目录结构与模块划分

#### 3.1.1 前端博客页面代码重复（P1）

**现状：** `BlogHome.vue`、`BlogCategory.vue`、`BlogTag.vue` 三个页面包含几乎相同的模板（文章卡片列表）、CSS（`.article-card`、`.card-title` 等 20+ 行样式）和逻辑（`formatDate`、`goDetail`、分页）。

**影响：** 修改文章卡片样式需要同步改 3 个文件，极易遗漏。

**改进方案：**

```
codenow-frontend/src/
├── components/
│   └── ArticleCard.vue          # 新增：文章卡片组件
├── composables/
│   └── useArticleList.js        # 新增：文章列表逻辑复用
└── utils/
    └── format.js                # 新增：formatDate 等工具函数
```

- 提取 `ArticleCard.vue` 组件，接收 `article` 和 `categoryName`、`tags` 作为 props
- 提取 `useArticleList()` 组合式函数，封装分页、筛选、数据加载逻辑
- 提取 `formatDate()` 到 `utils/format.js`

#### 3.1.2 前端缺少 API 层抽象（P1）

**现状：** API 路径字符串（如 `/blog/articles`、`/categories`）散落在 12+ 个组件中。

**影响：** 后端改接口路径时需逐个修改前端组件。

**改进方案：** 新建 `src/api/` 目录，按模块封装：

```javascript
// src/api/article.js
import request from '@/utils/request'
export const getArticles = (params) => request.get('/articles', { params })
export const getArticle = (id) => request.get(`/articles/${id}`)
export const createArticle = (data) => request.post('/articles', data)
// ...
```

#### 3.1.3 前端缺少 404 路由（P2）

**现状：** 访问不存在的路径会显示空白页。

**改进方案：** 在 `router/index.js` 末尾添加通配路由：

```javascript
{ path: '/:pathMatch(.*)*', redirect: '/blog' }
```

#### 3.1.4 PageQuery 死代码（P2）

**现状：** `common/PageQuery.java` 存在但无任何引用，所有分页使用 `@RequestParam`。

**改进方案：** 删除 `PageQuery.java`，或在 Controller 中统一使用。

#### 3.1.5 HotArticleService 架构不一致（P2）

**现状：** 其他 Service 都是接口 + Impl 模式，`HotArticleService` 是直接放在 `service/` 下的具体类。

**改进方案：** 移动到 `service/impl/HotArticleServiceImpl.java`，或保持现状（因为它是纯 Redis 操作，不涉及数据库）。

### 3.2 配置与环境管理

#### 3.2.1 配置文件硬编码密码（P0）

**现状：** `application.yaml` 中数据库密码 `123456` 和 Redis 密码 `123456` 直接写死并提交到 Git。

**影响：** 密码泄露到代码仓库，生产环境可能使用弱密码。

**改进方案：**

```yaml
# application.yaml
spring:
  datasource:
    password: ${DB_PASSWORD:123456}    # 开发环境用默认值，生产通过环境变量注入
  data:
    redis:
      password: ${REDIS_PASSWORD:123456}
```

#### 3.2.2 Sa-Token Redis 依赖被注释（P1）

**现状：** `pom.xml` 中 `sa-token-redis-jackson` 依赖被注释掉，Sa-Token 使用内存存储 Session。

**影响：** 生产环境多实例部署时，用户在 A 实例登录后，B 实例无法识别。

**改进方案：** 取消注释，在 `application-prod.yml` 中配置 Redis 连接。

#### 3.2.3 Token 发送无 Bearer 前缀（P1）

**现状：** `request.js` 发送 `Authorization: <token>`，无 `Bearer` 前缀。

**影响：** 不符合通用规范，对接第三方工具时可能不兼容。

**改进方案：**

```javascript
// request.js 请求拦截器
config.headers.Authorization = `Bearer ${token}`
```

同时在 Sa-Token 配置中设置 `token-prefix: Bearer`。

### 3.3 代码规范与工程化

#### 3.3.1 RedisConfig 反序列化漏洞（P0）

**现状：** `GenericJackson2JsonRedisSerializer` 使用 `LaissezFaireSubTypeValidator.instance`，允许反序列化任意类。

**影响：** 攻击者如果能写入 Redis，可以注入恶意对象实现远程代码执行。

**改进方案：**

```java
BasicPolymorphicTypeValidator ptv = BasicPolymorphicTypeValidator.builder()
    .allowIfSubType(Object.class)
    .allowIfSubType("com.codenow.")
    .allowIfSubType("java.util.")
    .allowIfSubType("java.time.")
    .build();
om.activateDefaultTyping(ptv, ObjectMapper.DefaultTyping.NON_FINAL);
```

#### 3.3.2 RateLimit Lua 脚本 Bug（P0）

**现状：** `PEXPIRE` 使用 `ARGV[3] * 2000`，但 `ARGV[3]` 是 `maxCount`（请求数），不是时间窗口。`@RateLimit(maxCount=5, timeWindow=60)` 实际 10 秒后 Key 就过期。

**影响：** 限流功能形同虚设。

**改进方案：** 修改 Lua 脚本，增加 `ARGV[4]` 传入时间窗口秒数：

```lua
-- rate_limit.lua
redis.call('PEXPIRE', KEYS[1], tonumber(ARGV[4]) * 1000)
```

同时修改 `RateLimitAspect.java` 传入 `timeWindow` 值。

#### 3.3.3 评论接口无认证（P0）

**现状：** `/api/comments/**` 全部排除在 Sa-Token 拦截器外，POST 评论无需登录且 `status=1` 自动通过。

**影响：** 垃圾评论泛滥，XSS 风险。

**改进方案：**
- 方案 A：POST 评论接口不从排除列表移除，但需要处理未登录用户发评论的场景（加验证码）
- 方案 B：默认 `status=0`（待审核），管理员在后台审核后才显示

#### 3.3.4 marked v18 代码高亮失效（P0）

**现状：** `BlogArticle.vue` 使用 `marked.setOptions({ highlight })`，但该 API 在 marked v5 中已移除。

**影响：** 文章详情页代码块无语法高亮。

**改进方案：**

```bash
npm install marked-highlight
```

```javascript
import { marked } from 'marked'
import { markedHighlight } from 'marked-highlight'
import hljs from 'highlight.js'

marked.use(markedHighlight({
  langPrefix: 'hljs language-',
  highlight(code, lang) {
    return hljs.highlightAuto(code).value
  }
}))
```

#### 3.3.5 getClientIp() 重复实现（P1）

**现状：** 在 `OperationLogAspect`、`RateLimitAspect`、`CommentController` 中各有一份，逻辑不一致。

**改进方案：** 提取到 `common/IpUtils.java`：

```java
public class IpUtils {
    public static String getClientIp(HttpServletRequest request) { ... }
}
```

#### 3.3.6 BCryptPasswordEncoder 应为 Spring Bean（P1）

**现状：** `AuthController` 中 `new BCryptPasswordEncoder()` 硬编码。

**改进方案：** 在 `SecurityConfig` 中注册 Bean，通过构造器注入使用。

#### 3.3.7 GlobalExceptionHandler 泄露堆栈（P1）

**现状：** 通用 Exception 处理返回 `e.getMessage()`，可能暴露 SQL 错误、类名等。

**改进方案：**

```java
@ExceptionHandler(Exception.class)
public R<Void> handleException(Exception e) {
    log.error("系统异常", e);
    return R.error("系统繁忙，请稍后再试");
}
```

#### 3.3.8 Element Plus 全量导入（P1）

**现状：** `app.use(ElementPlus)` 导入全部组件。

**改进方案：** 使用 `unplugin-vue-components` + `unplugin-auto-import` 实现按需导入：

```bash
npm install -D unplugin-vue-components unplugin-auto-import
```

```javascript
// vite.config.js
import AutoImport from 'unplugin-auto-import/vite'
import Components from 'unplugin-vue-components/vite'
import { ElementPlusResolver } from 'unplugin-vue-components/resolvers'

export default defineConfig({
  plugins: [
    vue(),
    AutoImport({ resolvers: [ElementPlusResolver()] }),
    Components({ resolvers: [ElementPlusResolver()] }),
  ],
})
```

#### 3.3.9 魔法数字（P2）

**现状：** `status=0/1`、`isTop=0/1` 在代码中直接使用数字。

**改进方案：** 定义枚举常量：

```java
public interface ArticleStatus {
    int DRAFT = 0;
    int PUBLISHED = 1;
}
```

### 3.4 部署与运维

#### 3.4.1 后端 Dockerfile 非多阶段构建（P1）

**现状：** Dockerfile 只复制预构建的 JAR，需要宿主机安装 JDK + Maven。

**改进方案：** 改为多阶段构建：

```dockerfile
# 构建阶段
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /app
COPY pom.xml .
COPY .mvn .mvn
COPY mvnw .
RUN ./mvnw dependency:go-offline -B
COPY src ./src
RUN ./mvnw clean package -DskipTests -B

# 运行阶段
FROM eclipse-temurin:21-jre-alpine
# ... 后续不变
COPY --from=builder /app/target/codenow-backend.jar app.jar
```

#### 3.4.2 添加 .dockerignore（P1）

**现状：** 构建时 `node_modules/` 和 `target/` 全部发送到 Docker daemon。

**改进方案：** 在 `codenow-backend/` 和 `codenow-frontend/` 各创建 `.dockerignore`：

```
# codenow-backend/.dockerignore
target/
.git
.idea
*.md
```

```
# codenow-frontend/.dockerignore
node_modules/
dist/
.git
*.md
```

#### 3.4.3 docker-compose 添加资源限制和日志轮转（P1）

**改进方案：**

```yaml
services:
  mysql:
    # ... 其他配置
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '0.5'
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  backend:
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '0.5'
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

#### 3.4.4 Nginx 添加缓存头和安全头（P2）

**改进方案：**

```nginx
# 静态资源缓存
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
    expires 7d;
    add_header Cache-Control "public, immutable";
}

# 安全头
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
```

#### 3.4.5 CI/CD 配置（P2）

建议添加 GitHub Actions 配置：

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - run: cd codenow-backend && ./mvnw test
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - run: cd codenow-frontend && npm ci && npm run build
```

### 3.5 可扩展性与维护成本

#### 3.5.1 零测试覆盖（P1）

**现状：** 后端仅有一个空的 `contextLoads()` 测试，前端无任何测试。

**改进方案：** 优先为核心 Service 层编写单元测试：

| 测试目标 | 测试内容 |
|---------|---------|
| `BlogArticleServiceImpl` | 文章 CRUD、标签关联、VO 组装 |
| `CommentServiceImpl` | 评论树组装、递归删除 |
| `AuthController` | 登录成功/失败、Token 生成 |
| `RateLimitAspect` | 限流放行/拒绝 |

#### 3.5.2 SysUser.role 未使用（P2）

**现状：** `role` 字段存在但 Sa-Token 只检查 `isLogin()`，不区分角色。

**改进方案：** 后续如需权限控制，使用 Sa-Token 的 `@SaCheckRole` 注解：

```java
@SaCheckRole("admin")
@DeleteMapping("/{id}")
public R<Void> delete(@PathVariable Long id) { ... }
```

## 五、验收标准

### P0 修复验收

- [ ] Redis 中存储的对象使用白名单反序列化，不使用 `LaissezFaireSubTypeValidator`
- [ ] `@RateLimit(maxCount=5, timeWindow=60)` 在 60 秒窗口内超过 5 次后正确拒绝
- [ ] 未登录用户 POST 评论被拦截或评论状态为待审核
- [ ] `application.yaml` 中无硬编码密码，全部使用 `${ENV_VAR:default}` 格式
- [ ] 文章详情页代码块有语法高亮显示

### P1 修复验收

- [ ] Sa-Token Session 存储在 Redis 中
- [ ] `docker compose build` 不需要宿主机安装 JDK
- [ ] 构建上下文不包含 `node_modules/` 和 `target/`
- [ ] `docker compose up` 后各容器有内存限制
- [ ] 前端 `BlogHome`、`BlogCategory`、`BlogTag` 使用共享的 `ArticleCard` 组件
- [ ] 前端 API 路径集中在 `src/api/` 目录管理
- [ ] `getClientIp()` 只有一个实现在 `IpUtils` 中
- [ ] `npm run build` 后 JS 包体积明显减小（Element Plus 按需导入）
- [ ] 生产环境异常响应不包含堆栈信息
- [ ] 核心 Service 有单元测试覆盖

### 整体验收

- [ ] `mvnw compile` 编译无错误
- [ ] `npm run build` 构建无错误
- [ ] `docker compose up -d` 一键启动全部服务
- [ ] 浏览器访问 `http://localhost/blog` 博客前台正常
- [ ] 浏览器访问 `http://localhost` 管理后台正常
