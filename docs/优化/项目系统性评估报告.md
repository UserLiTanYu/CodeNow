# 码上记（CodeNow）项目系统性评估报告

> 评估日期：2026-07-14 | 评估范围：全栈代码库 + 部署配置 + 数据库设计

---

## 一、代码质量与健壮性

### 1.1 关键问题

| # | 严重性 | 问题 | 位置 | 影响 |
|---|--------|------|------|------|
| 1 | **关键** | CORS 允许所有来源 + 携带凭证 | `CorsConfig.java` | 任意网站可发起跨域认证请求，若 Sa-Token 使用 Cookie 存储则可劫持会话 |
| 2 | **关键** | X-Forwarded-For 可伪造，绕过限流 | `RateLimitAspect.java` | 攻击者伪造 IP 头即可绕过滑动窗口限流 |
| 3 | **高** | 浏览量并发更新存在竞态条件 | `BlogArticleServiceImpl.java` | 先读后写模式导致 Redis 分数与数据库值漂移 |
| 4 | **高** | 评论内容/昵称未做服务端 XSS 过滤 | `CommentController.java` | 存储型 XSS：恶意 HTML 存入数据库后在管理后台执行 |
| 5 | **高** | 文件上传仅校验 Content-Type 头 | `UploadController.java` | 可伪造为 image/jpeg 上传 HTML/SVG 文件，OSS 直接以 HTML 渲染导致 XSS |
| 6 | **高** | BlogController.hotArticles 存在 N+1 查询 | `BlogController.java` | 10 篇热门文章触发 30 条 SQL（每篇 3 次查询） |
| 7 | **中** | AuthController.me() 未做 null 检查 | `AuthController.java` | 用户被删除但 Token 仍有效时触发 NPE |
| 8 | **中** | toggleStatus/toggleTop 存在 TOCTOU 竞态 | `ArticleController.java` | 并发切换可能丢失状态变更 |
| 9 | **中** | 评论接口未校验 articleId 是否存在 | `CommentController.java` | 可向不存在的文章提交评论，产生孤立数据 |
| 10 | **中** | AsyncConfig 未配置线程池 | `AsyncConfig.java` | SimpleAsyncTaskExecutor 每次创建新线程，高并发下线程数无上限 |

### 1.2 数据一致性问题

| # | 严重性 | 问题 | 位置 |
|---|--------|------|------|
| 1 | **高** | 文章软删除但标签关联硬删除，恢复文章后标签永久丢失 | `BlogArticleServiceImpl.java` |
| 2 | **高** | 浏览量 SQL 原子递增正确，但读取旧值推送到 Redis 导致数据漂移 | `BlogArticleServiceImpl.java` |
| 3 | **中** | 评论递归删除在中途崩溃时可能留下部分孤立子评论 | `CommentServiceImpl.java` |
| 4 | **中** | 文章更新时先删后插标签关联，但未校验 tagId 是否存在 | `BlogArticleServiceImpl.java` |

### 1.3 代码坏味道

| 问题 | 位置 | 建议 |
|------|------|------|
| `getClientIp()` 在 3 处重复实现且逻辑不一致 | RateLimitAspect / CommentController / OperationLogAspect | 提取为 `IpUtils.getClientIp(request)` 工具类 |
| 魔法数字（status=1, parentId=0L）散落多处 | 多个文件 | 定义常量 `ArticleStatus.PUBLISHED`、`CommentStatus.ROOT_PARENT_ID` |
| pageArticleVO 与 pagePublishedArticles 存在重复的标签筛选逻辑 | `BlogArticleServiceImpl.java` | 抽取为私有方法 `filterByTagId()` |
| HotArticleServiceImpl 吞掉所有异常返回空值 | `HotArticleServiceImpl.java` | Redis 故障时应抛出异常或返回降级标识 |
| OSS 初始化失败被静默吞掉 | `OssStorageServiceImpl.java` | 应在 `@PostConstruct` 中 fail-fast |
| 全局异常处理器始终返回 HTTP 200 | `GlobalExceptionHandler.java` | 应为 404/400/405 设置对应的 HTTP 状态码 |

### 1.4 改进方案

**并发安全修复（浏览量）：**
```java
// 改为：先更新数据库，再重新读取真实值推送到 Redis
lambdaUpdate().eq(BlogArticle::getId, id)
    .setSql("view_count = view_count + 1").update();
// 重新读取
int actualViewCount = getById(id).getViewCount();
hotArticleService.incrementViewCount(id, actualViewCount);
```

**CORS 限制：**
```java
config.addAllowedOriginPattern("http://localhost:*");
// 生产环境替换为实际域名
```

**IP 获取安全化：**
仅在 `getRemoteAddr()` 来自可信代理时才信任 `X-Forwarded-For`，否则直接使用 `getRemoteAddr()`。

## 二、架构设计与可扩展性

### 2.1 服务分层评估

| 层 | 评价 |
|----|------|
| Controller | 职责清晰，但 ArticleController 中 `toggleStatus`/`toggleTop` 包含业务逻辑，应下沉到 Service |
| Service | `BlogArticleServiceImpl` 承载过多职责（CRUD + 标签关联 + VO 组装 + 浏览量 + Redis），建议拆分 |
| Mapper | 纯 MyBatis-Plus BaseMapper 继承，无自定义 SQL，简洁合理 |
| DTO/VO | 分离合理，但 `ArticleVO` 包装方式（article + categoryName + tags）增加了序列化复杂度 |

### 2.2 前端状态管理评估

| 问题 | 影响 | 建议 |
|------|------|------|
| 仅有一个 userStore，其余状态全在组件内 | 博客侧边栏每次导航都重新请求分类/标签/热门文章 | 新增 `blogStore` 缓存侧边栏数据 |
| BlogCategory.vue 和 BlogTag.vue 各自重新请求分类/标签列表只为解析一个名称 | 浪费 API 调用 | 从 blogStore 读取 |
| 路由守卫直接读 localStorage，与 userStore.token 不同步 | Token 状态可能不一致 | 守卫改用 `useUserStore().token` |

### 2.3 前端组件复用性评估

| 问题 | 影响 |
|------|------|
| BlogHome / BlogCategory / BlogTag 三个页面大量重复模板和 CSS（~200 行） | 修改需同步改 3 个文件 |
| CommentTree.vue 中 `formatTime` 与 `formatDate` 工具函数重复 | 代码冗余 |
| CommentForm 使用原生 HTML 元素，与其余页面的 Element Plus 风格不一致 | UI 不统一 |
| ImageUpload 缺少客户端文件大小/类型校验 | 用户体验差，浪费带宽 |

### 2.4 技术选型评估

| 技术 | 评价 |
|------|------|
| **Sa-Token** | 轻量易用，但 UUID Token 不可签名，建议生产环境切换为 JWT 模式 |
| **MyBatis-Plus 3.5.7** | 与 Spring Boot 3.4.7 兼容性良好，但需验证最新版本安全补丁 |
| **Element Plus** | 按需导入已配置，但 main.js 仍全量导入 CSS（~400KB），需移除 |
| **highlight.js** | 全量导入 ~200 种语言语法，应改为 core + 按需导入 |
| **marked + md-editor-v3** | 两套 Markdown 渲染管线并存（marked 用于只读，md-editor-v3 用于编辑），可统一 |
| **Spring Security** | 仅用于 BCrypt，但引入了整个 Filter Chain 攻击面，可替换为 `spring-security-crypto` |

## 三、性能与可运维性

### 3.1 潜在性能瓶颈

| # | 严重性 | 问题 | 位置 | 影响 |
|---|--------|------|------|------|
| 1 | **高** | hotArticles 接口 N+1 查询（10 篇 = 30 条 SQL） | `BlogController.java` | 热门文章接口响应慢 |
| 2 | **中** | 评论树无分页，文章有数千条评论时一次性加载全部 | `CommentServiceImpl.java` | 内存压力 + 响应慢 |
| 3 | **中** | 前端 highlight.js 全量打包（~1MB） | `BlogArticle.vue` | 首屏加载慢 |
| 4 | **中** | 前端 Element Plus CSS 全量导入（~400KB） | `main.js` | 首屏加载慢 |
| 5 | **低** | 标签关联逐条插入而非批量 | `BlogArticleServiceImpl.java` | 批量操作效率低 |

### 3.2 监控与可观测性

| 维度 | 现状 | 评价 |
|------|------|------|
| 日志 | SLF4J + Logback，开发环境 StdOut，生产环境无文件输出 | 生产环境需配置日志文件滚动 |
| 指标 | 无 Micrometer/Prometheus 集成 | 无法监控 QPS、延迟、错误率 |
| 链路追踪 | 无 Sleuth/Zipkin 集成 | 分布式场景无法定位慢请求 |
| 健康检查 | 仅 docker-compose 级别的 MySQL/Redis 健康检查 | 后端无 `/actuator/health` 端点 |

| # | 严重性 | 问题 | 位置 |
|---|--------|------|------|

| 2 | **关键** | docker-compose 所有服务默认密码 `123456` | `docker-compose.yml` |
| 3 | **高** | 容器以 root 用户运行（无 non-root USER） | 两个 Dockerfile |

## 五、问题汇总与优先级

### 关键（立即修复）

| # | 问题 | 修复方案 |
|---|------|---------|
| 1 | `.env` 含密码已提交到 Git | `git rm --cached .env`，轮换所有密码 |
| 2 | CORS 允许所有来源 + 凭证 | 限制为实际前端域名 |
| 3 | X-Forwarded-For 可伪造绕过限流 | 仅信任可信代理的 IP，否则用 `getRemoteAddr()` |
| 4 | docker-compose 默认密码 123456 | 移除默认值，强制通过 .env 配置 |
| 5 | 前端 v-html + marked 未做 XSS 过滤 | 使用 DOMPurify 或 marked 的 sanitizer 选项 |
| 6 | 文件上传 Content-Type 可伪造 | 校验文件 magic bytes 而非依赖 header |

### 高优先级

| # | 问题 | 修复方案 |
|---|------|---------|
| 7 | 浏览量竞态条件导致 Redis 分数漂移 | 更新后重新读取真实值推送到 Redis |

| 9 | 容器以 root 运行 | Dockerfile 添加 non-root USER |

| 11 | 缺少 Content-Security-Policy 头 | nginx.conf 添加 CSP |
