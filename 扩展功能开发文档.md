## 一、Redis 缓存热门文章

> 难度标签：`简单快速` | 预估工时：1 天

### 1.1 功能简介

将浏览量最高的 Top 10 文章缓存到 Redis，博客首页的"热门文章"直接从 Redis 读取，不再每次都查数据库。

> 🍔 **生活类比**：就像餐厅把最热门的几道菜提前做好放在窗口，客人点了直接端走，不用每次都去厨房现做。

### 1.2 求职亮点

- **缓存策略**：面试必问"你们项目用了缓存吗？"，这个功能让你有实际项目可以讲
- **数据结构选型**：可以聊为什么选 ZSet（有序集合）而不是 String，以及 Score 用浏览量排序的好处
- **缓存一致性**：文章浏览量更新时，缓存怎么同步？可以聊 Cache-Aside 模式（先更新 DB，再删缓存）
- **简历话术**：「引入 Redis 缓存热门文章，采用 ZSet 有序集合按浏览量排序，配合 TTL 过期机制保证数据时效性，有效降低数据库查询压力」

### 1.3 实现思路

**核心流程：**

1. 文章被访问时，同步更新 MySQL 的 `view_count` 和 Redis ZSet 的 Score
2. 博客首页请求热门文章时，先从 Redis ZSet 取 Top 10 文章 ID
3. 再根据 ID 列表批量查 MySQL 拿到完整数据（ZSet 只存 ID，不存完整数据）
4. 设置 TTL（如 5 分钟），过期后自动从数据库重建

**数据结构：**

```
Redis Key:   codenow:hot_articles
Type:        ZSet
Score:       view_count（浏览量）
Member:      articleId（文章 ID）
TTL:         300 秒（5 分钟）
```

**涉及技术点：**

- Spring Data Redis（`StringRedisTemplate` 或 `RedisTemplate`）
- ZSet 操作：`zAdd`（更新分数）、`zRevRange`（取 Top N）、`expire`（设置 TTL）
- Cache-Aside 缓存模式

### 1.4 后端实现要点

| 层 | 改动 |
|----|------|
| **Config** | 新增 `RedisConfig.java`，配置 `RedisTemplate<String, Object>` 序列化（JSON） |
| **Service** | 新增 `HotArticleService.java`，封装缓存操作：`refreshHotArticles()`、`getHotArticleIds()`、`incrementViewCount(articleId)` |
| **Service** | 修改 `BlogArticleServiceImpl.getPublishedArticleById()`，访问文章时同步更新 Redis ZSet 的 Score |
| **Controller** | 修改 `BlogController`，新增 `GET /api/blog/articles/hot` 接口返回热门文章列表 |

**数据库变更：** 无（复用现有的 `view_count` 字段）

### 1.5 前端实现要点

| 页面/组件 | 改动 |
|-----------|------|
| `BlogLayout.vue` | 侧边栏新增"热门文章"模块，调用 `/api/blog/articles/hot` |
| `BlogHome.vue` | 可选：首页顶部增加"热门推荐"区域 |

### 1.6 验收标准

- [ ] 访问一篇文章后，Redis ZSet 中该文章的 Score 更新为最新浏览量
- [ ] 调用 `/api/blog/articles/hot` 返回浏览量 Top 10 的已发布文章

## 二、操作日志（AOP 实现）

> 难度标签：`中等投入` | 预估工时：2 天

### 2.1 功能简介

管理员在后台的每一次操作（登录、发布文章、删除分类等）都自动记录到日志表中，可在管理端查看操作历史。

> 📹 **生活类比**：就像银行的交易流水，每一笔存取款都自动记录时间、金额、操作人，出问题时可以追溯。

### 2.2 求职亮点

- **AOP 切面编程**：Spring 面试高频考点，这个功能让你从"知道 AOP 概念"升级到"实际用 AOP 解决过问题"
- **自定义注解**：可以聊怎么通过自定义注解 `@OperationLog` 标记需要记录的方法，比写死方法名优雅得多
- **异步处理**：日志写入可以聊 `@Async` 异步化，不阻塞主业务流程
- **元数据获取**：通过反射获取方法名、参数、IP 地址等，可以聊 Java 反射机制
- **简历话术**：「基于 AOP 实现操作日志记录，通过自定义注解标记需记录的方法，异步写入数据库，支持记录操作人、操作类型、请求参数、IP 地址等信息」

### 2.3 实现思路

**核心流程：**

1. 定义 `@OperationLog` 注解，标注在 Controller 方法上，描述操作类型
2. 定义 AOP 切面，拦截所有带 `@OperationLog` 注解的方法
3. 在 `@Around` 通知中：记录方法名、参数、返回值、耗时、操作人、IP 地址
4. 通过 `@Async` 异步将日志写入数据库
5. 管理端新增"操作日志"页面，分页查看日志列表

**数据库变更：**

新增 `sys_operation_log` 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| user_id | BIGINT | 操作人 ID |
| username | VARCHAR(50) | 操作人用户名 |
| operation | VARCHAR(100) | 操作描述（如"发布文章"） |
| method | VARCHAR(200) | 请求方法（如 `ArticleController.save`） |
| params | TEXT | 请求参数 JSON |
| ip | VARCHAR(50) | 请求 IP |
| duration | INT | 耗时（毫秒） |
| status | TINYINT | 状态（0=失败, 1=成功） |
| create_time | DATETIME | 创建时间 |

### 2.4 后端实现要点

| 层 | 改动 |

| **Entity** | 新增 `SysOperationLog.java` 实体类 |
| **Mapper** | 新增 `SysOperationLogMapper.java` |
| **Service** | 新增 `OperationLogService.java` + Impl |

## 三、评论系统

> 难度标签：`中等投入` | 预估工时：3 天

### 3.1 功能简介

在博客前台的文章详情页下方，访客可以发表评论，支持楼中楼回复（一级评论 → 二级回复 → 三级回复……）。

> 💬 **生活类比**：就像微信群聊，有人发了一条消息（一级评论），其他人可以"引用回复"（二级评论），还可以继续回复引用的消息（三级评论），形成对话树。

### 3.2 求职亮点

- **树形数据设计**：面试常问"怎么存评论这种树形结构？"，这个功能让你有实际方案可以讲
- **自关联表设计**：`parent_id` 指向自身表的 `id`，是最经典的树形存储方案
- **递归查询/渲染**：可以聊数据库递归查询（MySQL 8.0 CTE）或前端递归渲染组件
- **DTO 设计**：评论列表需要嵌套结构，可以聊 VO 组装和递归转换
- **简历话术**：「设计并实现评论系统，采用自关联表存储树形评论结构，支持多级嵌套回复，通过递归算法组装评论树返回前端」

### 3.3 实现思路

**核心流程：**

1. 用户在文章详情页发表评论（可选回复某条评论）
2. 评论存入数据库，`parent_id` 指向被回复的评论（顶级评论 `parent_id = 0`）
3. 查询评论时，一次性取出该文章的所有评论，在内存中组装成树形结构
4. 前端递归渲染评论树

**数据库变更：**

新增 `blog_comment` 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| article_id | BIGINT | 文章 ID |
| parent_id | BIGINT | 父评论 ID（顶级评论为 0） |
| content | TEXT | 评论内容 |
| nickname | VARCHAR(50) | 昵称（访客填写） |
| email | VARCHAR(100) | 邮箱（可选，不公开显示） |
| ip | VARCHAR(50) | 评论者 IP |
| status | TINYINT | 状态（0=待审核, 1=已通过） |
| create_time | DATETIME | 创建时间 |

**组装评论树的算法：**

```
1. 查出该文章所有评论（按 create_time ASC 排序）
2. 遍历评论列表，用 Map<parentId, List<Comment>> 分组
3. 从 parentId=0 的顶级评论开始，递归填充 children 列表
4. 返回树形结构给前端
```

| **Mapper** | 新增 `BlogCommentMapper.java` |
| **Service** | 新增 `CommentService.java` + Impl，包含 `getCommentTree(articleId)` 和 `saveComment()` |

### 3.5 前端实现要点

| 页面/组件 | 改动 |
|-----------|------|

- [ ] 发表一级评论后，`blog_comment` 表插入记录，`parent_id = 0`

## 四、API 接口限流

> 难度标签：`中等投入` | 预估工时：1.5 天

### 4.1 功能简介

对博客公开 API（如文章列表、文章详情）做访问频率限制，同一 IP 每秒最多请求 10 次，超过则返回 429 状态码。

> 🚦 **生活类比**：就像高速公路的收费站，每秒只放行 10 辆车，后面的车必须排队等待，防止道路被堵死。

### 4.2 求职亮点

- **Redis + Lua 脚本**：面试常问"限流怎么实现？"，Redis + Lua 是最经典的方案之一
- **滑动窗口算法**：可以聊固定窗口 vs 滑动窗口 vs 漏桶 vs 令牌桶的区别和适用场景
- **自定义注解 + AOP**：通过 `@RateLimit` 注解标记需要限流的方法，参数可配置
- **防御性编程**：体现你对系统安全和稳定性的考虑
- **简历话术**：「基于 Redis + Lua 实现滑动窗口限流，通过自定义注解标记需限流的接口，支持按 IP 维度配置限流规则，有效防止恶意刷接口」

### 4.3 实现思路

**核心流程：**

1. 定义 `@RateLimit` 注解，配置限流参数（如 `maxRequests=10, windowSeconds=1`）
2. AOP 切面拦截带 `@RateLimit` 注解的方法
3. 执行 Lua 脚本：用 Redis Sorted Set 记录请求时间戳，判断窗口内请求数是否超限
4. 超限则抛出异常，返回 429 状态码

**Lua 脚本逻辑（滑动窗口）：**

```lua
-- KEYS[1]: 限流 Key（如 rate_limit:/api/blog/articles:127.0.0.1）
-- ARGV[1]: 窗口起始时间（当前时间 - 窗口秒数）
-- ARGV[2]: 当前时间戳
-- ARGV[3]: 最大请求数

-- 1. 删除窗口外的旧记录
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[1])
-- 2. 统计窗口内的请求数
local count = redis.call('ZCARD', KEYS[1])
-- 3. 判断是否超限
if count < tonumber(ARGV[3]) then
    -- 未超限，记录本次请求
    redis.call('ZADD', KEYS[1], ARGV[2], ARGV[2])
    redis.call('EXPIRE', KEYS[1], tonumber(ARGV[3]))
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

### 4.4 后端实现要点

| 层 | 改动 |
|----|------|
| **Annotation** | 新增 `@RateLimit` 注解（`maxRequests`、`windowSeconds`、`message` 属性） |
| **Aspect** | 新增 `RateLimitAspect.java`，读取注解参数，执行 Lua 脚本判断 |
| **Config** | 新增 `RedisConfig.java`（如果还没有），配置 `DefaultRedisScript` 加载 Lua 脚本 |
| **Resource** | 新增 `scripts/rate_limit.lua` Lua 脚本文件 |
| **Controller** | 在 `BlogController` 的公开接口上添加 `@RateLimit(maxRequests = 10, windowSeconds = 1)` |
| **Exception** | `GlobalExceptionHandler` 新增限流异常处理，返回 429 状态码 |

**数据库变更：** 无（限流数据存在 Redis 中）

### 4.5 前端实现要点

| 页面/组件 | 改动 |
|-----------|------|
| `utils/request.js` | 响应拦截器中增加 429 状态码处理，提示"请求过于频繁，请稍后再试" |

改动量极小，主要工作在后端。

### 4.6 验收标准

- [ ] `@RateLimit` 注解可配置 `maxRequests`（最大请求数）和 `windowSeconds`（窗口秒数）
- [ ] 快速连续请求超过限制后，返回 `{code: 429, message: "请求过于频繁"}` 
- [ ] 等待窗口时间后，请求恢复正常
- [ ] 限流数据存储在 Redis 中，Key 格式为 `rate_limit:{接口}:{IP}`
- [ ] 面试时能画出滑动窗口算法示意图，说清楚和固定窗口的区别

## 五、文件上传（封面图 / 富文本图片）

> 难度标签：`中等投入` | 预估工时：2 天

### 5.1 功能简介

文章编辑时可直接上传封面图和内容图片，不再需要手填 URL。支持本地存储和阿里云 OSS 两种方案。

> 📷 **生活类比**：就像发朋友圈时直接拍照上传，而不是先传到图床再复制链接粘贴。

### 5.2 求职亮点

- **Spring MultipartFile**：文件上传是 Web 开发基础技能，面试偶尔会问
- **策略模式**：本地存储和 OSS 存储可以聊策略模式（`StorageStrategy` 接口 + 多实现）
- **文件类型校验**：安全角度考虑，需要校验文件类型和大小，可以聊防御性编程
- **对象存储**：对接 OSS 可以聊云服务集成、预签名 URL、CDN 加速
- **简历话术**：「实现文件上传功能，采用策略模式支持本地存储和阿里云 OSS 两种方案，通过文件类型白名单和大小限制保障上传安全」

### 5.3 实现思路

**核心流程：**

1. 前端选择文件，通过 FormData 上传到后端
2. 后端校验文件类型（仅允许 jpg/png/gif/webp）和大小（限制 5MB）
3. 生成唯一文件名（UUID + 原始扩展名），避免重名
4. 存储到指定目录（本地）或 OSS Bucket
5. 返回文件访问 URL，前端回显

**数据库变更：** 无（文件 URL 存在文章表的 `cover_image` 或 `content` 字段中）

### 5.4 后端实现要点

| **Service** | 新增 `LocalStorageServiceImpl.java`（本地磁盘存储） |
| **Service** | 新增 `OssStorageServiceImpl.java`（阿里云 OSS 存储，可选） |

- [ ] `POST /api/upload/image` 可上传图片文件，返回可访问的 URL
- [ ] 上传非图片文件（如 .txt、.exe）被拒绝，返回错误提示
- [ ] 上传超过 5MB 的文件被拒绝
- [ ] 上传的图片可通过返回的 URL 直接访问
- [ ] 文章编辑页的封面图可通过上传组件选择图片

## 六、Docker 容器化部署

> 难度标签：`较大挑战` | 预估工时：3 天

### 6.1 功能简介

编写 Dockerfile 和 docker-compose.yml，一条命令启动整个系统：MySQL + Redis + 后端 + 前端 + Nginx 五个容器协同工作。

> 🧳 **生活类比**：就像搬家时把所有东西装进标准集装箱，不管搬到哪台服务器，只要装了 Docker，一条命令就能把整个系统"搬"过去运行。

### 6.2 求职亮点

- **Docker 基础**：容器化是现代开发的标配技能，面试几乎必问
- **docker-compose 编排**：多容器编排可以聊服务依赖、网络配置、数据持久化
- **Nginx 反向代理**：前端静态文件服务 + 后端 API 转发，可以聊反向代理和负载均衡
- **多阶段构建**：前端镜像用多阶段构建（Node 构建 + Nginx 运行），减小镜像体积
- **生产环境意识**：体现你不只会写代码，还能把项目跑起来
- **简历话术**：「使用 Docker Compose 实现一键部署，编排 MySQL、Redis、后端、前端、Nginx 五个服务，采用多阶段构建优化前端镜像体积，通过 Nginx 实现反向代理和静态资源服务」

### 6.3 实现思路

**整体架构：**

```
                    ┌──────────┐
                    │  Nginx   │ :80 / :443
                    │ (前端+代理)│
                    └────┬─────┘
                         │
              ┌──────────┴──────────┐
              │ /                   │ /api
              ▼                     ▼
        ┌──────────┐         ┌──────────┐
        │ 前端静态  │         │  后端API  │ :8080
        │  dist/   │         │  Spring  │
        └──────────┘         └─────┬────┘
                                   │
                         ┌─────────┴─────────┐
                         ▼                   ▼
                   ┌──────────┐        ┌──────────┐
                   │  MySQL   │        │  Redis   │
                   │  :3306   │        │  :6379   │
                   └──────────┘        └──────────┘
```

**需要创建的文件：**

```
codenow/
├── docker-compose.yml           # 编排文件
├── nginx/
│   └── nginx.conf               # Nginx 配置
├── codenow-backend/
│   ├── Dockerfile               # 后端镜像
│   └── ...
└── codenow-frontend/
    ├── Dockerfile               # 前端镜像（多阶段构建）
    └── ...
```

### 6.4 后端实现要点

| 文件 | 内容 |
|------|------|
| `codenow-backend/Dockerfile` | 基础镜像 `eclipse-temurin:21-jre`，复制 JAR 包，暴露 8080 端口 |
| `codenow-backend/pom.xml` | 确认 `spring-boot-maven-plugin` 可正常打包 |
| `application-prod.yml` | 生产环境配置（数据库连接改为容器名 `mysql`，Redis 改为 `redis`） |
| `docker-compose.yml` | 定义 5 个服务：mysql、redis、backend、frontend、nginx |

**docker-compose.yml 核心结构：**

```yaml
services:
  mysql:
    image: mysql:8.0
    environment: MYSQL_ROOT_PASSWORD, MYSQL_DATABASE
    volumes: mysql_data + init.sql

  redis:
    image: redis:7-alpine

  backend:
    build: ./codenow-backend
    depends_on: mysql, redis
    environment: spring.profiles.active=prod

  frontend:
    build: ./codenow-frontend
    # 多阶段构建：Node 编译 → Nginx 运行

  nginx:
    image: nginx:alpine
    ports: 80:80
    volumes: nginx.conf + 前端 dist
    depends_on: backend, frontend
```

### 6.5 前端实现要点

| 文件 | 内容 |
|------|------|
| `codenow-frontend/Dockerfile` | 多阶段构建：第一阶段 `node:22` 执行 `npm run build`，第二阶段 `nginx:alpine` 复制 dist |
| `vite.config.js` | 确认 API 代理配置在生产环境也能工作（或通过 Nginx 代理） |

**Nginx 配置核心：**

```nginx
server {
    listen 80;
    # 前端静态文件
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;  # Vue Router history 模式
    }
    # 后端 API 代理
    location /api/ {
        proxy_pass http://backend:8080;
    }
}
```

### 6.6 验收标准

- [ ] `docker-compose up -d` 一条命令启动全部 5 个服务
- [ ] `docker-compose ps` 显示所有容器状态为 Up
- [ ] 访问 `http://localhost` 能看到博客前台页面
- [ ] 访问 `http://localhost/api/blog/articles` 能返回文章数据
- [ ] MySQL 数据持久化（`docker-compose down` 后重新 `up`，数据不丢失）
- [ ] 后端日志可通过 `docker-compose logs backend` 查看
- [ ] 面试时能画出容器架构图，说清楚 Nginx 如何做反向代理
